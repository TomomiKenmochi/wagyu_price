\# Git Lesson



\## リモートリポジトリとローカルリポジトリとはそれぞれ何でしょうか？
 
  * リポジトリ  
     ファイルやディレクトリの状態を記録する場所のこと。
  * リモートリポジトリ  
     リポジトリがネット上に配置されているもの。複数人でソースコードを共有するのに最適。
  * ローカルリポジトリ  
     自分や各々のPC内にリポジトリが配置されているもの。手元のPCで開発をするのに最適。


\## コミットとプッシュの違い
 
 * コミット   
     バージョンの保存をすること。ファイルの編集を記録するコマンド。ゲームのセーブデータのようなもの。
 * プッシュ  
     ローカルでコミットした編集記録をリモートへ送るコマンドの事。ローカルリポジトリの編集を送りこみリモートリポジトリへ反映させる。



\## コミットのメッセージはどのように書いてあげるのが最適でしょうか？

  * 各チームやプロジェクトのルールに準じて、抽象的ではなく具体的に何をしたか誰が見てもわかるように書くことが最適である。  
      例）間違い訂正×　→　index.htmlのカード１列目の表示名間違いあり変更○



\## ローカルでマージするフローと、プルリクエストでマージするフローの違いは何でしょうか？
 
 * ローカルでマージするフロー  
      ローカルworking branch→リモートworking branchへpush  
      →ローカルworking branch→ローカルmaster branchへmerge  
      →**ローカルmaster branch→リモートmaster branchへpush**  
      　master branchへの反映がとても速い。欠点としては自分以外の誰の目が入らない為、重大なミスやバグがそのまま反映されて大きな損害や故障が起こる可能性がある事。


 * プルリクエストでマージするフロー  
     ローカルworking branch→リモートworking branchへpush  
     →（プルリクエストをレビュー担当者が確認）（OK後、リモートmaster branchへmergeしてもらう）  
     →**ローカルmaster branch→リモートmaster branchからpull**  
     　自分自身以外の誰かの目が入るため間違えが見つかりやすくバグやミスを無くしやすくできる。欠点としては自分自身以外に依頼、確認、mergeしてもらう必要がある為時間がかかる事。注意点はpullを忘れてしまうとローカルmaster branchは何も更新されない為、今後エラーが起き何もできなくなる可能性がある事。


  * **大きな違いは「品質維持（プルリクエストでマージ＞ローカルでマージ）」と「スピード（ローカルでマージ＞プルリクエストでマージ）」である。**


\## コンフリクトを起こしてしまった場合、どう対処すべきですか？

  * まず出てきたソースコードを読む。エディタを使って編集（選択）する。自分以外の人が書いたソースコードの内容を確実に把握する必要がある為、読み取れない処理がある場合は必ずソースコードを書いた人と相談をしながら進めていく。
